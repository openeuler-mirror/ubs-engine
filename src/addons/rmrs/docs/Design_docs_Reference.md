# Summary

UBS RMRS是一款面向虚拟化场景的节点内资源管理服务插件，作为UBS Engine框架的一部分提供集群级的内存资源调度服务，主要针对虚机内存碎片场景、虚机内存超分场景、容器内存超分场景。

虚机内存碎片场景下，旨在利用节点间搁浅内存构建超节点内存池多卖内存立即分配模式的虚机，UBS RMRS插件对外提供节点间反亲和配置接口、内存借用决策接口、内存借用执行接口、内存迁出决策接口、内存迁出执行接口、内存归还决策与执行接口、借用内存回滚接口。

虚机内存超分场景，旨在利用按需分配的虚机暂时用不到的内存实现虚机超卖，UBS RMRS插件对外提供内存借用接口、内存分配接口、内存归还接口。

容器内存超分场景，旨在利用容器暂时用不到的内存实现容器超创，UBS RMRS插件对外提供内存借用接口、内存分配接口、内存归还接口。

# Motivation

UBS RMRS是一款面向虚拟化场景的节点内资源管理服务插件，作为UBS Engine框架的一部分提供集群级的内存资源调度服务，主要针对虚机内存碎片场景、虚机内存超分场景、容器内存超分场景。

虚机碎片架构分为两层：

UBS RMRS部署在UBS engine内，各计算节点Agent端仅负责数据采集和消息转发等，主节点侧作为功能入口，提供碎片内存的借用、迁出，内存归还和回滚接口，负责节点间的碎片内存管理。

RMRS部署在各计算节点的OS Turbo内，基于自身的资源采集，提供碎片场景虚机的迁出/迁回策略，利用SMAP迁移能力执行虚机的迁出和迁回。

虚机/容器超分场景：

MemLink动态回收虚机的空余内存或补充内存。在numa内存触发高水位水线时，触发内存借用，并迁出虚机内存，缓解内存压力。在numa内存触发低水位水线时，归还借用内存。

UBS RMRS提供内存借用、虚机迁移、内存归还接口，提供虚机迁移算法。OS Turbo提供资源采集能力，供迁出算法决策。

# Detailed Design

虚机碎片场景提供以下能力：

1、内存借用

- 内存借用策略实现思路 

  虚机创建内存不足时触发OpenStack插件，调用该策略模块。

  内存借用策略思路：根据创建虚拟机所需内存，向邻居节点借用内存，保证在能借到足够内存的前提下尽量避免借出节点资源碎片化。同时保证借用内存节点能够置换出足够本地内存。

  借用决策结果默认按照4G切分单个动作，可通过修改配置去除此限制；

  注意：反亲和信息仅在本节点做借用决策和在中心侧做故障借用决策时生效；设置反亲和信息也仅保存在调用节点和中心侧，不在其他节点生效；

  按照优先级排序：

  1）遍历直连的邻居节点，获取可以借用的节点列表，考虑反亲和性和借用白名单(节点可以向哪些节点借入）；

  2）借用不能成环（借入节点不能作为借出节点，借出节点不能作为借入节点）；

  3）借用策略需要考虑UBS Engine框架 group和provider配置项，决定本节点可以向哪些直连邻居节点借入内存；

  4）默认只从直连的一跳的socket节点借用内存（如果配置为优先向同平面借用，则这里为优先向从直连的一跳的socket节点借用内存）；

  5）优先从已借入过的socket节点借入内存；

  6）优先从剩余可借用内存多的socket节点的借用；

  7）优先从可借用内存多的Numa节点借用内存；（确认是否有多个Numa);

- 内存借用执行实现思路借用策略需要考虑UBS Engine框架group和provider配置项，决定本节点可以向哪些直连邻居节点借入内存；内存借用执行思路：内存借用执行动作会改变集群状态，因此要保证动作的原子性。遍历执行内存借用动作，当执行到某个动作失败时，应当把已经执行成功的部分回滚。考虑以下几种情况：

  默认仅能向同平面的socket节点借内存；如果配置为优先向同平面借用，则没有这个参数校验限制了；

  借用执行动作默认单个借用动作不超过4G，可通过修改配置去除此限制；单个执行动作集不超过1T，超时时间为1h；

  内存借用执行是中心侧（主节点）控制的，因此每次执行内存借用前会在主节点持久化当前执行的动作并同步到备节点：

  1）执行动作集全部借用动作执行成功，删除所有主、备上持久化保存的执行动作；

  2）执行到某个动作失败，删除已成功借入的内存（以资源标识符borrowId为粒度），每删除一条则对应删除持久化的借用动作并进行数据同步；

  3）执行到某个动作时主节点断连，当选出新主节点时会等待内存子系统发布的就绪事件，当触发内存子系统就绪事件后读取在新主节点（原备节点）上持久化的借用动作，然后依次调用内存归并删除对应持久化的本条借用动作;

2、内存迁出

- 内存迁出策略实现思路 

  创建虚拟机本地内存不足，需匀出本地内存时，调用该策略模块。

  内存迁出思路：根据需要匀出本地内存大小、虚拟机对应迁出最大比例、远端NUMA内存空闲情况、虚拟机迁出情况综合决策哪些虚拟机迁出到哪些远端NUMA上多少内存和预计执行迁出动作集合所需时间。

  具体步骤如下：

  1）校验输入虚拟机pid和对应最大迁出比例是否有效；

  2）获取借入节点的numa信息，判断远端空闲内存是否满足匀出本地内存大小，满足则进行步骤3；

  3）获取借入节点的vm信息，判断输入vm列表还可以迁出的内存大小，是否满足匀出本地内存大小，满足则进行步骤4）；

  4）对虚拟机还可以迁出的内存从大到小排序（已经迁出/远近端比例创建的虚拟机排在内存都在本地的虚拟机之前）；

  5）采用深度优先搜索算法决策虚拟机迁出到哪些远端numa上和迁出多少内存，虚拟机所占用本地numa的socket和迁到远端numa的借出节点的socket是在同一个平面的(根据配置项可配置，只能迁出到同平面或者优先迁出到同平面)；

  6）预测虚拟机迁出内存需要等待时间；

- 内存迁出执行实现思路根据入参在指定节点迁出指定的一批虚机到指定远端numa上指定的大小，并记录入参中借用记录标识与虚机标识、迁出大小的关联。

​		注意，迁出执行入参中需要传入超时时间（waitTime），表示单个执行动作不能超过此时间，否则认为执行失		败；当waitTime传0时，认为没有超时时间约束，同步等待迁移执行动作返回结果，仅作用于大虚机场景；

​		1）校验入参中借用标识、节点标识、迁出的远端numa、虚机标识是否有效，不能存在重复的虚机标识，校验虚		拟机是否迁出到相同平面借用过来的内存上(根据配置项可配置，只能迁出到同平面或者优先迁出到同平面)；

​		2）存储入参中借用标识和虚机标识、虚机远端迁出大小的映射关系，回滚接口使用

​		3）记录入参中的迁出节点、迁出的虚机以及迁出的远端numa，被动备升主中断迁出流程重启过程会将这些迁出		的虚机重新迁回。

​		4）发送rpc到迁出的节点执行实际的迁出决策和执行逻辑

​		根据入参查询指定的numa、虚机信息，判断远端numa大页空余量是否能够满足根据入参迁出虚机指定大小全部		迁出，假如虚机远端已经存在内存占用，迁出时判断远端余量时会考虑到已经占用的部分，只考虑增量迁出的部		分远端是否有空余的大页。

​		满足余量判断后持久化迁出执行的入参，后续如果因UBS engine进程崩溃导致的迁出中断会根据记录的数据进行		回滚

​		迁出执行前记录迁出虚机在远端numa占用的原始大小，如果迁出过程失败则回滚已执行迁出虚机在远端的大小		至记录的原始大小（执行前会在内存中记录虚拟机在远端Numa上占用的内存大小）

​		如果虚机迁出到远端大小为0或者回滚成0，则将该虚机从smap的管理对象中删除，SMAP不再对该虚机进行冷热		扫描

​		5）全部执行完清空持久化文件的执行入参，删除记录的用于主备流程中断的入参记录

虚机/容器超分场景提供以下能力：

1、支持内存借用，生成borrowId，根据borrowId来申请资源借用内存调UBS Engine中numa借用接口（ubse会进行借用决策、执行）申请资源借用内存
  虚机超分场景下：传入高水线numa对应的socketId及需要借用大小列表，ubse会借入直连邻居节点同一平面的远端内存
  容器超分场景下：1.容器绑定单个本地numa（绑cpu核）场景，传入高水线numa对应的socketId及需要借用大小列表，ubse会借入直连邻居节点同一平面的远端内存
                2.容器不绑定numa场景，传入socketId（此时填充为-1）及需要借用大小列表，此时ubse会借入直连邻居节点可借入内存（不区分平面）

2、支持内存迁出

- 内存超分场景下的内存迁出分为以下几个步骤：

  1. 上层触发水线，调用超分场景内存迁出功能
  2. 触发内存借用
  3. 借用成功，触发内存迁出

  

  迁移策略如下：

  1. 按借用配额对本节点远端numa上可迁出内存的容器进行降序排序，优先使用借用配额大的
  2. 优先从存在远端使用内存（已迁出过）的虚机进行分配
  3. 筛选已使用远端内存的虚机
  4. 计算它们的剩余可迁出内存 = 总内存*比例 - 已迁出内存
  5. 对这些虚机按可迁出内存进行降序排序
  6. 根据排序结果，按剩余可迁出内存从大到小，分配迁出内存
  7. 若2步骤迁出内存不够，则从未使用过远端内存的虚拟机中分配
  8. 筛选未使用远端内存的虚机
  9. 计算它们的剩余可迁出内存 = 总内存*比例
  10. 对这些虚机按可迁出内存进行降序排序
  11. 根据排序结果，按剩余可迁出内存从大到小，分配迁出内存
  12. 若完成3步后仍有空闲的内存且远端numa数量大于1则通过深度优先算法决策已在远端numa的虚机在远端numa之间进行调度，保证远端numa内存最大占用算法步骤如下：

  ​    4.1 初始化检查：

  ​    计算理想的空余内存（远端numa内存 - pid总内存*迁出比例）

  ​    检查所有VM是否已分配。如果是，更新最佳情况并返回。

  ​    通过计算当前状态的理想空余值，若理想空余值都无法达到目前最佳的情况直接跳过当前状态，如果是，直接返回。

  ​    4.2 遍历每个VM：

  ​    对于每个VM，如果已分配，跳过。

  ​    标记VM为已分配。

  ​    4.3 处理原始资源：

  ​    获取VM的原始资源位置。如果存在，释放该资源的内存。

  ​    4.4 尝试迁移至其他资源：

  ​    遍历所有资源，检查是否有足够的剩余容量容纳当前VM。

  ​    如果有，备份资源容量，减少该资源的容量，记录迁移步骤，并递归调用DFS处理下一个VM。

  ​    4.5 回溯步骤：

  ​    递归返回后，恢复资源容量和VM的位置，回退迁移步骤。

  ​    4.6 恢复原始状态：

  ​    恢复VM的原始资源位置和容量，标记VM为未分配，继续处理下一个VM。

  算法若超过500ms直接终止

3、支持内存归还

- 1.更新内存账本并获取内存账本：通过内存子系统接口更新账本信息，如果更新获取失败则归还失败，成功则从获取到的内存账本中提取borrowId借用大小map，borrowId对应numa借用大小map，borrowId中使用的远端numaId map。

  2.获取迁出的pid：通过rpc调用从Smap中获取对应numa远端容器pid信息

  3.判断borrowId对应numa借用内存大小map或 borrowId中使用的远端numaId map是否为空: 若有一个为空则说明对应的borrowId已经归还内存了，无需再次归还；若都不为空，则进入归还逻辑。

  4.发送rpc消息向agent获取借入方节点本地的虚拟机信息：通过借入方本地numa上的虚机和取到的借出方远端numa过滤出告警节点对应numa上的进程

  5.过滤borrowID：若远端numa上有正在热迁移的虚机则该远端numa上内存不能归还

  6.开始执行单个borrowId的归还

  7.设置告警numa远端numa使用内存上限:通过rpc调用smap设置告警numa远端numa使用内存上限（**SMAP归还内存大小，borrowID需要乘以分配比例**）

  8.开始内存归还/释放资源：设置告警numa远端numa使用内存上限成功后开始执行内存归还

  9.根据borrowId来释放资源：若归还失败

  9.1 如果失败但是没有对obmm设备进行操作，则对告警numa远端numa使用内存上限进行回滚重置。

  9.2 否则就调用强制归还进行删除。

  10.若释放资源返回成功则判断远端numa内存是否清空。

  11.清除迁出进程冷热页扫描：若远端numa上已经没有内存使用则通过rpc调用smap来清除迁出进程冷热页扫描。

# Design constraints

- 用户使用UBTurbo进行内存借用时，需保证迁出内存地址和迁入内存地址的安全性一致。
- 用户管理的需要迁移的虚机或容器对应的用户权限应该和远端内存对应的用户权限保持一致。
- 客户使用UBTurbo，需要将用户添加到UBTurbo属组，被添加用户须拥有节点内存资源管理员的权限，才能使用UBTurbo内存迁移的能力，在内存迁移中PID由集群资源管理中心管理和下发，UBTurbo组件无法校验PID有效性，需要开发者在整体解决方案中，综合考虑pid、srcNid，destNid等参数传输和存储的安全。 

- 应用无需做特殊更改
