# CLOS双层选主对账与状态同步设计

## 1. 背景

CLOS场景下存在两层选主关系：

- 机柜内选主：每个机柜产生一个机柜主，负责本机柜节点信息汇聚、节点对账、内存账本修复。
- 全局选主：PD机柜主之间产生一个全局主，负责跨机柜视角下的节点状态汇聚和跨机柜调度可见性控制。

内存借用调度只允许使用同时满足以下条件的节点：

- `clusterState == UBSE_NODE_WORKING`
- `globalState == UBSE_NODE_GLOBAL_READY`

因此，全局主切换后不能仅依赖节点状态上报就把节点放回调度池。必须先确认该节点的全局账本摘要已经上报到当前全局主，之后才能把 `globalState` 从 `INIT` 推进到 `READY`。

## 2. 设计目标

1. 全局主上线后，所有节点的 `globalState` 先回到 `UBSE_NODE_GLOBAL_INIT`。
2. 节点账本摘要未上报到全局主前，`globalState` 保持 `INIT`。
3. 节点账本摘要已上报后，再由 node 层推进 `globalState` 到 `READY`。
4. 状态变更边界保持在 node 层，mem 层只提供账本摘要是否存在的门禁能力。
5. 全局主上线事件既要覆盖 PD 主，也要覆盖挂载在 PD 下的机柜主。
6. 空账本也视为有效上报，只要该节点在摘要存储中存在记录即可。

## 3. 模块职责

### 3.1 node controller

node controller 是节点状态的唯一推进方，负责：

- 处理 HA 事件，包括主上线、全局主上线、节点上下线。
- 维护 `UbseNodeInfo.clusterState` 和 `UbseNodeInfo.globalState`。
- 接收机柜汇聚上报、全局汇聚上报、单节点状态上报。
- 在全局主侧执行 `ProcessGlobalStateAfterReport()`，根据回调结果推进 `globalState`。
- 将全局主上线事件转换为本地事件，并通知挂载机柜主。

关键接口：

- `UbseGlobalMasterOnlineHandler()`
- `ProcessGlobalStateAfterReport()`
- `UpdateNodeInfoGlobalState()`
- `ResetAllGlobalStates()`
- `UbseMasterNotifyMountedGroupMastersAction()`

### 3.2 mem controller

mem controller 负责内存账本和账本摘要，避免直接修改 node 的全局状态：

- 节点进入 `SMOOTHING` 后执行内存账本对账。
- 对账成功后生成并上报全局账本摘要。
- 全局主上线本地事件触发后，为本机柜内已有 `WORKING` 节点重新上报摘要。
- 注册 `RegGlobalStateNotifyHandler()` 回调，检查目标节点是否已经上报过账本摘要。

关键接口：

- `LedgerHandler()`
- `SubmitNodeLedgerSummary()`
- `ReportExistingSummaryForWorkingNode()`
- `StoreGlobalNodeLedgerSummary()`
- `HasStoredGlobalNodeLedgerSummary()`
- `CheckGlobalLedgerSummaryReported()`

### 3.3 summary store

`UbseGlobalLedgerSummaryStore` 是全局主侧的账本摘要存储：

- 按 `nodeId` 保存最近一次摘要。
- 保存上报来源机柜主、落库时间戳。
- 提供 `ContainsNodeSummary()`，用于判断某节点是否已经上报过摘要。
- `ContainsNodeSummary()` 不检查摘要 item 数量，因此空账本上报也会返回 true。

## 4. 核心数据

### 4.1 节点状态

`UbseNodeInfo` 中和本流程相关的字段：

- `clusterState`：机柜主侧节点状态，例如 `WORKING`、`SMOOTHING`、`FAULT`、`UNKNOWN`。
- `globalState`：全局主侧对账同步状态，目前关键状态为 `INIT` 和 `READY`。

`globalState` 的推进规则：

- 全局主上线或切换时，批量重置为 `INIT`。
- 普通节点状态上报不会直接推进到 `READY`。
- 全局主收到节点状态上报后，执行 global notify handlers。
- 所有回调成功后，node 层调用 `UpdateNodeInfoGlobalState(nodeId, READY)`。
- 已经 `READY` 的节点不会被普通更新降级到 `INIT`，降级必须走 `ResetAllGlobalStates()`。

### 4.2 账本摘要

`UbseGlobalNodeLedgerSummary` 包含：

- `nodeId`：摘要所属节点。
- `sourceMasterNodeId`：生成摘要的机柜主。
- `reportTimestampMs`：全局主落库时间。
- `shmSummary`：共享内存导入/导出摘要。

当前摘要内容覆盖共享内存账本，已销毁状态的对象不会进入摘要。

## 5. 全局主上线流程

### 5.1 HA 事件入口

node controller 注册 `GLOBAL_MASTER_ONLINE_NOTIFICATION`。收到事件后进入：

```text
UbseGlobalMasterOnlineHandler(globalMasterId)
```

处理步骤：

1. 打印当前节点和角色。
2. 调用 `ResetAllGlobalStates()`，所有节点 `globalState` 重置为 `INIT`。
3. 发布本地事件 `UBSE_EVENT_GLOBAL_MASTER_ONLINE`。
4. 如果当前节点是 `PD_MASTER` 或 `GLOBAL_MASTER`，调用 `UbseMasterNotifyMountedGroupMastersAction()` 通知挂载机柜主。

### 5.2 为什么需要通知挂载机柜主

`GLOBAL_MASTER_ONLINE_NOTIFICATION` 只会发给 PD 侧主节点。挂载机柜主收不到该 HA 事件，但它持有本机柜节点的账本上下文，必须在全局主上线时重新上报已有 `WORKING` 节点的摘要。

因此由 PD 主/全局主通过 node controller 的 `NODE_CONTROLLER_NODE_CHANGE` RPC 通知挂载机柜主。挂载机柜主收到 `UBSE_EVENT_GLOBAL_MASTER_ONLINE` action 后，只发布本地事件，不触发 URMA 拓扑更新。

### 5.3 通知目标

挂载机柜主通知目标来自：

```text
UbseElectionModule::GetPdMountedGroupMasters()
```

不使用 `nextId`，因为全局主的 `nextId` 可能包含 PD 主，不能精确表示“挂载在当前 PD 下的机柜主”。

## 6. 摘要重新上报流程

mem controller 在初始化时订阅本地事件：

```text
UBSE_EVENT_GLOBAL_MASTER_ONLINE -> GlobalMasterOnlineEventHandler()
```

事件处理步骤：

1. 调用 `ClearStoredGlobalNodeLedgerSummaries()` 清空本地摘要存储。
2. 遍历 node controller 缓存中的所有节点。
3. 对 `clusterState == WORKING` 的节点投递异步任务。
4. 异步任务调用 `ReportExistingSummaryForWorkingNode(nodeId)`。
5. 查询本机柜账本上下文，构造摘要。
6. 通过 `UBSE_MEM_GLOBAL_LEDGER_SUMMARY_REPORT` RPC 上报到当前全局主。
7. 全局主 mem RPC handler 调用 `StoreGlobalNodeLedgerSummary()` 落库。

注意：该流程不是重新执行完整账本对账，而是对当前已有账本上下文生成摘要并上报。完整账本对账仍由节点进入 `SMOOTHING` 后触发。

## 7. 节点对账与摘要上报流程

节点对账入口在 node controller：

```text
UbseNodeLedger(nodeId)
```

处理步骤：

1. node 层把 `clusterState` 更新为 `SMOOTHING`。
2. node 层调用 `ReportSingleNodeChangeToPrev()`，把 `SMOOTHING` 同步给上一层主。
3. mem 层 `LedgerHandler()` 收到 cluster state 回调，执行 FD、NUMA、Addr、Share 账本对账。
4. 对账成功后调用 `SubmitNodeLedgerSummary(nodeId)`。
5. 摘要上报到全局主并落库。
6. node 层把 `clusterState` 更新为 `WORKING`。
7. node 层再次调用 `ReportSingleNodeChangeToPrev()`，把 `WORKING` 同步给上一层主。

`ReportSingleNodeChangeToPrev()` 的转发规则：

- 当前节点是 `PD_MASTER`：向全局主上报单节点状态。
- 当前节点是挂载机柜主：向自己的 PD 主上报单节点状态。
- 当前节点是 `GLOBAL_MASTER`：不再向上转发。

## 8. globalState READY 门禁流程

全局主收到节点上报后，会进入：

```text
ProcessGlobalStateAfterReport(nodeId)
```

处理步骤：

1. 如果当前节点不是全局主，直接返回。
2. 查询 node controller 中缓存的 `UbseNodeInfo`。
3. 如果 `globalState` 已经是 `READY`，直接返回。
4. 执行 `ExecGlobalStateNotifyHandler(nodeInfo)`。
5. mem 注册的 `CheckGlobalLedgerSummaryReported(nodeInfo)` 检查 summary store 中是否存在该 `nodeId`。
6. 如果不存在，返回错误，`globalState` 保持 `INIT`。
7. 如果存在，node 层调用 `UpdateNodeInfoGlobalState(nodeId, READY)`。

这条路径保证了：

- 账本摘要未上报时，即使收到了节点状态，节点也不会进入全局可调度状态。
- 空账本上报后，summary store 中存在该 `nodeId`，门禁通过。
- 状态推进仍在 node 层完成，mem 不直接调用 `UpdateNodeInfoGlobalState()`。

## 9. 调度侧使用

mem scheduler 更新节点可用性时同时检查：

```text
clusterState == UBSE_NODE_WORKING &&
globalState == UBSE_NODE_GLOBAL_READY
```

因此全局主刚切换后，即使节点本身仍是 `WORKING`，只要摘要还没有重新上报并触发 `globalState READY`，该节点就不会参与新的借用调度。

## 10. 典型时序

### 10.1 全局主切换后已有 WORKING 节点恢复可调度

```text
HA -> PD主/全局主: GLOBAL_MASTER_ONLINE_NOTIFICATION
PD主/全局主 -> node: ResetAllGlobalStates()
PD主/全局主 -> local event: UBSE_EVENT_GLOBAL_MASTER_ONLINE
PD主/全局主 -> 挂载机柜主: NODE_CONTROLLER_NODE_CHANGE(UBSE_EVENT_GLOBAL_MASTER_ONLINE)
挂载机柜主 -> local event: UBSE_EVENT_GLOBAL_MASTER_ONLINE
各机柜主 mem -> 全局主 mem: UBSE_MEM_GLOBAL_LEDGER_SUMMARY_REPORT
全局主 mem -> summary store: StoreGlobalNodeLedgerSummary(nodeId)
机柜/PD状态上报 -> 全局主 node: ProcessGlobalStateAfterReport(nodeId)
全局主 node -> mem callback: CheckGlobalLedgerSummaryReported(nodeId)
全局主 node: UpdateNodeInfoGlobalState(nodeId, READY)
mem scheduler: 节点重新参与调度
```

### 10.2 新节点上线或节点重新对账

```text
节点上线/周期对账/故障恢复
机柜主 node: clusterState -> SMOOTHING
机柜主 mem: LedgerHandler() 执行账本对账
机柜主 mem -> 全局主 mem: 上报账本摘要
机柜主 node: clusterState -> WORKING
机柜主/PD主 -> 全局主 node: 上报节点状态
全局主 node: 摘要存在则 globalState -> READY
```

## 11. 异常处理

### 11.1 摘要未上报

`CheckGlobalLedgerSummaryReported()` 返回 `UBSE_ERR_NODE_NOT_EXIST`，`ProcessGlobalStateAfterReport()` 保持 `INIT`。后续节点状态再次上报时会重新尝试。

### 11.2 空账本

空账本摘要仍会调用 `StoreGlobalNodeLedgerSummary()`，summary store 中存在该 `nodeId`。因此 `HasStoredGlobalNodeLedgerSummary()` 返回 true，节点允许进入 `READY`。

### 11.3 摘要版本

当前 READY 门禁只需要判断“当前全局主周期内是否收到该节点摘要”。原有 `ledgerEpoch` 是节点/机柜主进程内存自增值，节点或进程重启后会重新计数，不能作为跨重启的拒绝依据，因此不再保留在摘要协议中。

### 11.4 全局主无法获取或 RPC 失败

摘要上报前会通过 election module 获取当前全局主。获取失败或 RPC 失败时，上报失败，summary store 不会新增该节点摘要，`globalState` 继续保持 `INIT`。

### 11.5 挂载机柜主通知失败

`UbseMasterNotifyMountedGroupMastersAction()` 使用异步 RPC 通知挂载机柜主。发送失败只打印告警，不阻断当前 PD 主本地处理。被漏通知的机柜主不会立即重新上报已有摘要，对应节点会保持 `INIT`，直到后续事件或状态上报触发补偿。

## 12. 边界约束

1. node 层负责状态推进，mem 层不直接修改 `globalState`。
2. mem 层通过 `RegGlobalStateNotifyHandler()` 提供 READY 前置检查。
3. 全局主上线传播使用 node 事件，不让 node 层直接调用 mem 接口。
4. `UbseMasterNotifyAllAgentsAction()` 仍只面向已有节点变更和拓扑变更，不改变原有语义。
5. `UBSE_EVENT_GLOBAL_MASTER_ONLINE` 在 agent 侧只发布本地事件，不调用 `SetUrmaUvs()`。

## 13. 当前限制与后续建议

1. 当前摘要只包含共享内存账本，FD、NUMA、Addr 的跨柜摘要能力可按需要扩展。
2. 挂载机柜主通知失败当前没有重试机制，建议后续增加事件补偿或周期性摘要重上报。
3. `globalState READY` 依赖后续节点状态上报触发。如果摘要先到但状态长时间不上报，READY 推进会延迟。可考虑摘要落库后由全局主主动触发一次对应节点的 global state 检查，但仍保持状态修改在 node 层。
