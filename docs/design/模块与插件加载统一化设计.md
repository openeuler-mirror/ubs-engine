# UBS Engine 模块与插件加载统一化设计

## 1 概述

### 1.1 背景

UBS Engine 作为集群/节点 UB 资源管理系统，内部包含多种功能模块，包括框架基础能力模块（配置、日志、事件、通信等）、业务控制器模块（内存池化、节点管理、Urma 控制器等）以及可扩展的增值插件（RMRS、VirtAgent、uCache 等）。原有架构中，存在以下一些问题

- 模块注册与加载方式不统一，模块与插件缺乏统一的分类管理机制，

- 插件无法以标准化方式将南向服务能力注册到 ubse 框架中供北向使用。

- 作为多资源管理系统，在不同环境中，并不是所有模块均需加载，需要根据场景变化，来加载不同模块

- 目前系统支持简单的场景切换来加载模块，但是是在原有基础宏上增加条件语句，例如`CONDITION_DYNAMIC_CREATE(GetSceneType() == SceneType::COMMON, UbseMmiModule);`

  嵌套在各自模块中，一个模块可能在多个场景中都需要加载；目前这种模式无法支持多场景按需加载；

  

### 1.2 设计目标

1. **统一模块分类**：将所有模块划分为核心模块（CORE）、可选模块（OPTIONAL）和插件模块（PLUGIN）三类，明确各类模块的定位与生命周期管理策略
2. **声明式注册**：通过宏实现模块的编译期自动注册，消除手动注册的遗漏风险
3. **依赖管理**：支持模块间依赖声明，框架自动进行拓扑排序，确保模块按依赖关系正确初始化与启动
4. **模块按需加载**：从原来的模块里嵌套场景，转换为根据场景需要,支持按需加载所需模块
5. **服务注册表**：提供统一的服务注册表（Service Registry），使南向插件能够将服务能力注册到 ubse 框架中，供 ubse 北向功能或其他模块发现与消费

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 核心模块（CORE） | 系统启动必须加载的模块，初始化失败将导致进程退出 |
| 可选模块（OPTIONAL） | 系统功能增强模块，受部署配置控制是否加载，加载失败不影响核心功能 |
| 插件模块（PLUGIN） | 通过动态库运行时加载的扩展模块，依赖可选模块 |
| 服务注册表（Service Registry） | 提供服务注册、发现与获取能力的全局单例，南向插件注册服务，ubse 北向消费服务 |

## 3 模块体系设计

### 3.1 模块基类 UbseModule

所有模块均需继承 `UbseModule` 基类，并实现以下生命周期接口：

```cpp
class UbseModule {
public:
    virtual ~UbseModule() = default;
    virtual uint32_t Initialize() = 0;
    virtual void UnInitialize() = 0;
    virtual uint32_t Start() = 0;
    virtual void Stop() = 0;
    virtual void RegArgs() {};
    virtual std::string Name() const = 0;
};
```

**生命周期说明：**

| 阶段 | 方法 | 说明 |
|------|------|------|
| 初始化 | `Initialize()` | 分配资源、读取配置、初始化内部组件 |
| 启动 | `Start()` | 启动业务逻辑、注册回调、开始对外服务 |
| 停止 | `Stop()` | 停止业务逻辑、注销回调、停止对外服务 |
| 反初始化 | `UnInitialize()` | 释放资源、清理内部状态 |
| 参数注册 | `RegArgs()` | 注册命令行参数（可选覆写） |

**约束：** 每个模块必须定义 `static constexpr const char* kModuleName` 作为模块唯一标识，用于注册表索引与依赖声明。

### 3.2 模块分类

#### 3.2.1 核心模块（CORE）

核心模块是系统运行的基础设施，在启动阶段必须全部成功加载与初始化，任一核心模块失败将导致进程退出。

**特征：**
- 编译期通过 `CORE_MODULE_IMPL` 宏自动注册
- 在 `UbseContext::Run` 的 Phase1 阶段优先创建与启动
- 停止时最后被销毁（逆序）

**现有核心模块示例：**

| 模块名 | 类名 | 依赖 |
|--------|------|------|
| UbseSecurityModule | UbseSecurityModule | 无 |
| UbseConfModule | UbseConfModule | UbseSecurityModule |
| UbseLoggerModule | UbseLoggerModule | UbseConfModule |
| UbseEventModule | UbseEventModule | UbseConfModule, UbseLoggerModule |
| UbseComModule | UbseComModule | UbseEventModule |
| UbseElectionModule | UbseElectionModule | UbseComModule |

#### 3.2.2 可选模块（OPTIONAL）

可选模块提供业务增强能力，受部署配置控制是否加载。

**特征：**

- 编译期通过 `OPTIONAL_MODULE_IMPL` 宏自动注册
- 在 Phase3 阶段（核心模块启动完成后）创建与启动
- 停止时先于核心模块被销毁

**现有可选模块示例：**

| 模块名 | 类名 | 依赖 |
|--------|------|------|
| UbsePluginModule | UbsePluginModule | UbseElectionModule |
| UbseUrmaControllerModule | UbseUrmaControllerModule | UbseNodeControllerModule, UbseElectionModule |
| UbseHttpModule | UbseHttpModule | UbseEventModule |
| UbseStorageModule | UbseStorageModule | UbseComModule |

#### 3.2.3 插件模块（PLUGIN）

插件模块通过动态库（`.so`）在运行时加载，是系统扩展性的核心载体。插件依赖可选模块。

**特征：**

- 运行时通过 `PLUGIN_MODULE_IMPL` 宏在动态库的 `__attribute__((constructor))` 中注册
- 依赖通过 `UbseOptionModule` 枚举声明，编译期类型检查
- 停止时最先被销毁，且会从注册表中移除

### 3.3 模块注册宏

#### 3.3.1 CORE_MODULE_IMPL

```cpp
#define CORE_MODULE_IMPL(CLASS, ...)
```

**功能：** 在匿名命名空间中定义一个 Registrar 结构体，利用静态对象的构造函数在 `main()` 执行前完成核心模块注册。

**参数：**
- `CLASS`：模块类名，需定义 `static constexpr const char* kModuleName`
- `...`（可变参数）：依赖模块的类型列表，通过 `Deps::kModuleName...` 展开为依赖名称

**使用示例：**

```cpp
// 无依赖
CORE_MODULE_IMPL(UbseSecurityModule);

// 依赖 UbseConfModule 和 UbseLoggerModule
CORE_MODULE_IMPL(UbseEventModule, UbseConfModule, UbseLoggerModule);
```

**注册机制：** 匿名命名空间中的静态对象在动态库加载时自动构造，构造函数调用 `UbseModuleRegistry::GetInstance().RegisterCoreModule<CLASS, Deps...>()`。

#### 3.3.2 OPTIONAL_MODULE_IMPL

```cpp
#define OPTIONAL_MODULE_IMPL(CLASS, ...)
```

**功能：** 与 `CORE_MODULE_IMPL` 机制相同，但注册类别为 `OPTIONAL`。

**使用示例：**

```cpp
OPTIONAL_MODULE_IMPL(UbsePluginModule, UbseElectionModule);
OPTIONAL_MODULE_IMPL(UbseUrmaControllerModule,
    ubse::nodeController::UbseNodeControllerModule, UbseElectionModule);
```

#### 3.3.3 PLUGIN_MODULE_IMPL

```cpp
#define PLUGIN_MODULE_IMPL(CLASS, MODULE_LIST)
```

**功能：** 使用 `extern "C"` 和 `__attribute__((constructor))` 在动态库加载时自动执行注册，确保插件被 `dlopen` 后立即注册到模块注册表。

**参数：**
- `CLASS`：插件模块类名
- `MODULE_LIST`：依赖模块的枚举容器（如 `std::array<UbseOptionModule, 2>{...}`），通过 `UbseModuleIdToString` 转换为字符串

**与 CORE/OPTIONAL 宏的差异：**

| 特性 | CORE/OPTIONAL | PLUGIN |
|------|---------------|--------|
| 注册时机 | 静态初始化阶段（`main` 前） | `dlopen` 时（`constructor` 属性） |
| 依赖声明方式 | 类型列表（`Deps::kModuleName...`） | 枚举容器（`UbseOptionModule`） |
| 依赖解析 | 编译期类型检查 | 运行时字符串匹配 |
| 链接方式 | 静态链接到主程序 | 动态库运行时加载 |

**使用示例：**

```cpp
PLUGIN_MODULE_IMPL(MyPluginModule,
    std::array<UbseOptionModule, 2>{
        UbseOptionModule::UbseComModule,
        UbseOptionModule::UbseElectionModule
    });
```

### 3.4 UbseOptionModule 枚举

插件模块的依赖通过 `UbseOptionModule` 枚举声明，提供编译期类型安全：

```cpp
enum class UbseOptionModule {
    UbseMmiModule,
    UbseLcneModule,
    SysSentryModule,
    UbseUrmaUvsModule,
    UbseNodeControllerModule,
    UbseUrmaControllerModule,
    UbseComModule,
    UbseElectionModule,
    UbseHttpModule,
    UbseStorageModule,
    UbseRasModule,
    UbseNodeMgrModule,
    UbsePluginModule,
};
```

`UbseModuleIdToString` 函数将枚举值转换为模块名称字符串，用于与注册表中的模块名匹配。

### 3.5 UbseModuleRegistry

`UbseModuleRegistry` 是全局模块注册表，采用 Meyer's Singleton 模式：

```cpp
class UbseModuleRegistry {
public:
    static UbseModuleRegistry &GetInstance();

    template <typename T, typename... Deps>
    void RegisterCoreModule();

    template <typename T, typename... Deps>
    void RegisterOptionalModule();

    template <typename T, typename Container>
    void RegisterPluginModule(const Container &deps);

    void UnregisterPluginModules();
    std::map<std::string, UbseModuleEntry> TakeRegistry();

private:
    UbseModuleRegistry() = default;
    std::map<std::string, UbseModuleEntry> registry_{};
};
```

**关键设计决策：**

1. **TakeRegistry 语义**：`TakeRegistry()` 通过 `std::move` 返回注册表，意味着注册表只能被消费一次。`UbseContext` 在 `CreateCoreModules` 时首次调用，获取 CORE/OPTIONAL 模块；在 `CreatePluginModules` 时再次调用，获取 PLUGIN 模块
2. **防重复注册**：`RegisterXxxModule` 方法在注册前检查模块名是否已存在，避免重复注册
3. **空名称保护**：`kModuleName` 为空时直接返回，不进行注册

### 3.6 UbseModuleEntry

```cpp
struct UbseModuleEntry {
    std::function<std::shared_ptr<UbseModule>()> creator;
    std::vector<std::string> dependencies;
    UbseModuleCategory category;
};
```

| 字段 | 说明 |
|------|------|
| `creator` | 模块工厂函数，延迟创建模块实例 |
| `dependencies` | 依赖模块名称列表 |
| `category` | 模块分类（CORE / OPTIONAL / PLUGIN） |

## 4 生命周期编排

### 4.1 启动流程

`UbseContext::Run` 按以下三个阶段编排模块启动：

```
┌─────────────────────────────────────────────────────────────┐
│ Phase 1: 核心模块加载                                        │
│                                                             │
│  1. TakeRegistry() → 获取 CORE + OPTIONAL 注册信息          │
│  2. ResolveActivation(CORE) → 激活所有 CORE 模块            │
│  3. TopologicalSort() → 拓扑排序                            │
│  4. 创建模块实例（按排序顺序）                                │
│  5. InitCoreModules() → 逐个调用 Initialize()               │
│  6. StartCoreModules() → 逐个调用 Start()                   │
├─────────────────────────────────────────────────────────────┤
│ Phase 2: 插件扫描                                           │
│                                                             │
│  UbsePluginLoader::DiscoverAndLoad()                        │
│  ├── 扫描 /usr/lib64/ubse_plugin/ 目录                     │
│  ├── 准入过滤（白名单机制）                                  │
│  └── dlopen() 加载 .so → 触发 constructor 注册 PLUGIN 模块  │
├─────────────────────────────────────────────────────────────┤
│ Phase 3: 非核心模块加载                                      │
│                                                             │
│  1. TakeRegistry() → 获取 PLUGIN 注册信息                   │
│  2. ResolveActivation(PLUGIN) → 激活 PLUGIN 模块及依赖      │
│  3. TopologicalSort() → 拓扑排序                            │
│  4. 创建模块实例（按排序顺序）                                │
│  5. InitNonCoreModules() → 逐个调用 Initialize()            │
│  6. StartNonCoreModules() → 逐个调用 Start()                │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 停止流程

`UbseContext::Stop` 按启动的逆序停止模块：

```
1. StopNonCoreModules()      → 逆序调用 PLUGIN + OPTIONAL 的 Stop()
2. DestroyNonCoreModules()   → 逆序调用 PLUGIN + OPTIONAL 的 UnInitialize()
   └── PLUGIN 模块额外从 registry_ 和 activated_ 中移除
3. UbsePluginLoader::UnloadAll() → dlclose() 卸载所有插件动态库
4. StopCoreModules()         → 逆序调用 CORE 的 Stop()
5. DestroyCoreModules()      → 逆序调用 CORE 的 UnInitialize()
6. 清理 sortedModules_, registry_, activated_, moduleMap_
```

### 4.3 依赖解析与拓扑排序

**激活规则（ResolveActivation）：**

- CORE 类别：所有已注册的 CORE 模块无条件激活
- PLUGIN 类别：仅激活已注册的 PLUGIN 模块，并递归激活其依赖链
- 依赖缺失的模块不会被激活，但不会导致启动失败

**拓扑排序（TopologicalSort）：**

基于 `activated_` 集合中的模块及其依赖关系，执行深度优先搜索（DFS）拓扑排序，确保被依赖模块排在依赖方之前。

## 5 插件加载机制

### 5.1 插件发现

`UbsePluginLoader` 负责在指定目录中扫描符合规则的动态库文件：

**发现规则：**
1. 插件目录必须存在，不存在则跳过（非错误）
2. 插件文件必须以 `lib` 开头、`.so` 结尾
3. 通过准入过滤后，使用 `dlopen(path, RTLD_LAZY | RTLD_NODELETE)` 加载

### 5.2 准入控制

`UbsePluginAdmission` 提供插件白名单机制：

| 场景 | 行为 |
|------|------|
| 准入配置加载成功且白名单非空 | 仅加载白名单中的插件 |
| 准入配置加载失败或白名单为空 | 仅加载默认插件 `mem_plugin` |

### 5.3 插件注册流程

```
dlopen(libxxx_plugin.so)
    │
    ▼ __attribute__((constructor))
XXX_plugin_init()
    │
    ▼
UbseModuleRegistry::GetInstance().RegisterPluginModule<XXX>(deps)
    │
    ▼ 写入 registry_
[等待 UbseContext::CreatePluginModules 消费]
```

### 5.4 插件卸载

1. `UbseContext::Stop` 调用 `DestroyNonCoreModules()`，逆序调用 PLUGIN 模块的 `UnInitialize()` 并从注册表移除
2. `UbsePluginLoader::UnloadAll()` 对所有 `dlopen` 句柄执行 `dlclose()`
3. `UbseModuleRegistry::UnregisterPluginModules()` 清理残留的 PLUGIN 注册信息

## 6 服务注册表设计

### 6.1 设计动机

模块体系解决了"如何加载与启动"的问题，但未解决"南向插件如何向 ubse 提供服务能力"的问题。插件作为南向扩展，加载后需要一种标准化的方式将自身能力注册到 ubse 框架中，供 ubse 北向功能（API Server、核心模块、可选模块等）发现与消费。服务注册表（`UbseServiceRegistry`）正是为此设计：

- **南向北向解耦**：南向插件作为服务提供者，ubse 北向功能作为服务消费者，双方通过服务名称解耦，无需直接依赖具体实现
- **类型安全**：通过模板与 `static_assert` 确保服务类型必须继承 `UbseIService`
- **线程安全**：使用 `std::shared_mutex` 实现读写锁，支持高并发读取
- **弱引用返回**：`GetService` 返回 `std::weak_ptr`，避免服务注册表持有强引用导致生命周期问题

### 6.2 UbseIService 接口

```cpp
class UbseIService {
public:
    virtual ~UbseIService() = default;
    virtual std::string GetServiceName() const = 0;
};
```

所有需要注册到服务注册表的服务必须继承 `UbseIService`，并实现 `GetServiceName()` 返回唯一服务名称。

### 6.3 UbseServiceRegistry API

```cpp
class UbseServiceRegistry {
public:
    static UbseServiceRegistry &GetInstance();

    template <typename T>
    void RegisterService(std::shared_ptr<T> service);

    template <typename T>
    void UnRegisterService(std::shared_ptr<T> service);

    template <typename T>
    std::weak_ptr<T> GetService(const std::string &name);

    bool HasService(const std::string &name) const;
};
```

| 方法 | 说明 |
|------|------|
| `RegisterService<T>` | 注册服务实例，T 必须继承 `UbseIService`，以 `GetServiceName()` 为键 |
| `UnRegisterService<T>` | 注销服务实例，以 `GetServiceName()` 为键移除 |
| `GetService<T>` | 按名称获取服务，返回 `weak_ptr<T>`，需调用方 `lock()` 后使用 |
| `HasService` | 检查服务是否存在，返回 `bool` |

### 6.4 线程安全策略

| 操作 | 锁类型 | 说明 |
|------|--------|------|
| `RegisterService` | `std::unique_lock`（写锁） | 修改 `services_` 映射 |
| `UnRegisterService` | `std::unique_lock`（写锁） | 修改 `services_` 映射 |
| `GetService` | `std::shared_lock`（读锁） | 只读访问 `services_` 映射 |
| `HasService` | `std::shared_lock`（读锁） | 只读访问 `services_` 映射 |

### 6.5 使用示例

**服务定义：**

```cpp
class IMyService : public UbseIService {
public:
    virtual void DoSomething() = 0;
};

class MyServiceImpl : public IMyService {
public:
    std::string GetServiceName() const override { return "MyService"; }
    void DoSomething() override { /* ... */ }
};
```

**服务注册（在插件模块Start 中，南向插件向 ubse 注册服务）：**

```cpp
UbseResult MyPluginModule::Start() {
    auto service = std::make_shared<MyServiceImpl>();
    UbseServiceRegistry::GetInstance().RegisterService(service);
    return UBSE_OK;
}
```

**服务消费（在 ubse 北向功能中，如核心模块或 API Server 获取插件服务）：**

```cpp
void UbseCoreModule::OnStart() {
    auto svc = UbseServiceRegistry::GetInstance().GetService<IMyService>("MyService");
    if (auto ptr = svc.lock()) {
        ptr->DoSomething();
    }
}
```

**服务注销（在插件模块 UnInitialize 中）：**

```cpp
void MyPluginModule::Stop() {

    UbseServiceRegistry::GetInstance().UnRegisterService(service);
}
```

### 6.6 模块体系与服务注册表的协作

```
┌─────────────────────────────────────────────────────────┐
│              ubse 北向功能层（服务消费者）                 │
│   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐   │
│   │  API Server  │ │ Core Module  │ │Optional Module│   │
│   └──────┬───────┘ └──────┬───────┘ └──────┬───────┘   │
│          │                │                │            │
│          └────────────────┼────────────────┘            │
│                           │ GetService<T>()             │
├───────────────────────────┼─────────────────────────────┤
│                           ▼                             │
│                  UbseServiceRegistry                     │
│                           ▲                             │
│                           │ RegisterService()           │
├───────────────────────────┼─────────────────────────────┤
│              南向插件层（服务提供者）                      │
│   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐   │
│   │  RMRS Plugin │ │VirtAgent     │ │ uCache Plugin│   │
│  │  └──────────────┘ └──────────────┘ └──────────────┘   │
└─────────────────────────────────────────────────────────┘
```

## 7 扩展指南

### 7.1 新增核心模块

1. 继承 `UbseModule`，定义 `static constexpr const char* kModuleName`
2. 实现生命周期接口（`Initialize`, `Start`, `Stop`, `UnInitialize`）
3. 在 `.cpp` 文件中使用 `CORE_MODULE_IMPL(ClassName, Dep1, Dep2, ...)` 注册
4. 将模块编译为静态库，链接到主程序

### 7.2 新增可选模块

步骤与核心模块相同，仅将宏替换为 `OPTIONAL_MODULE_IMPL`。

### 7.3 新增插件模块

1. 继承 `UbseModule`，定义 `static constexpr const char* kModuleName`
2. 实现生命周期接口
3. 在 `.cpp` 文件中使用 `PLUGIN_MODULE_IMPL(ClassName, deps)` 注册
4. 将模块编译为共享库（`.so`），输出到 `/usr/lib64/ubse_plugin/`
5. 如需准入控制，在准入配置中添加插件名称

### 7.4 新增服务

1. 在南向插件中定义服务接口，继承 `UbseIService`，声明业务方法
2. 实现服务类，覆写 `GetServiceName()` 返回唯一名称
3. 在插件模块 `Start()` 中创建服务实例并调用 `RegisterService`，将服务注册到 ubse 框架
4. 在插件模块 `Stop()` 中调用 `UnRegisterService`，注销服务
5. ubse 北向功能（核心模块、可选模块、API Server 等）通过 `GetService<T>(name).lock()` 获取插件服务实例

## 8 约束与注意事项

1. **模块名唯一性**：`kModuleName` 必须全局唯一，重复注册会被静默忽略
2. **依赖完整性**：模块声明的依赖必须已注册，否则该模块不会被激活
3. **循环依赖**：拓扑排序算法会检测循环依赖并抛出异常，模块设计应避免循环依赖
4. **线程安全**：`UbseServiceRegistry` 的注册操作应在南向插件初始化阶段执行，注销操作应在插件反初始化阶段执行，避免与 ubse 北向的 `GetService` 并发
5. **weak_ptr 使用**：ubse 北向消费者获取 `weak_ptr` 后应立即 `lock()` 并判空，不应长期持有 `weak_ptr` 假设南向插件服务始终可用；插件卸载后服务将不可访问
6. **插件动态库符号**：主程序编译时需添加 `-rdynamic -pie` 链接选项，确保插件动态库可以访问主程序的全局符号
7. **插件卸载时序**：插件卸载前必须确保所有异步操作已完成，避免 `dlclose` 后回调访问已卸载的代码段
